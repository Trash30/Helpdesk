PRÉREQUIS CÔTÉ DÉVELOPPEUR
À faire sur ton poste, AVANT le jour J

0.1 — Merger les PRs dans le bon ordre

Tu as 2 PRs ouvertes sur GitHub. Elles doivent être mergées dans cet ordre strict :

1
Étape 1 — Merger PR #2 en premier
URL : https://github.com/Trash30/Helpdesk/pull/2
Titre : feat: motif obligatoire PENDING + exclusion stale
Action : clique « Merge pull request » → « Confirm merge »
2
Étape 2 — Rebaser PR #1 sur le nouveau main, puis merger
Après le merge de PR #2, PR #1 a divergé. Il faut la mettre à jour : 
git checkout feat/client-organisation-tickettype
git pull origin main
git push origin feat/client-organisation-tickettype
Ensuite sur GitHub, PR #1 → « Merge pull request » → « Confirm merge »
Pourquoi cet ordre ? PR #2 corrige un bug sur les tickets PENDING. Si tu merges PR #1 en premier, ce bug revient en prod.

0.2 — Vérifier que main est propre après les merges

git checkout main
git pull origin main
git log --oneline -5

Tu dois voir les commits des deux PRs en tête de main. Si ce n'est pas le cas, STOP.

0.3 — Vérifier qu'il n'y a pas de fichiers de migrations Prisma

ls server/prisma/migrations/
IMPORTANT : Si ce dossier est vide (comme c'est le cas actuellement), deploy.sh va échouer à l'étape migrations. Voir la correction en Phase 2 étape 2.4.

INSTALLATION
Première fois uniquement

2.1 — Se connecter au serveur

ssh root@<IP_SERVEUR>

2.2 — Cloner le dépôt

git clone https://github.com/Trash30/Helpdesk.git /opt/helpdesk
cd /opt/helpdesk
git checkout main

2.3 — Lancer l'installateur

sudo bash install.sh

Le script va te poser des questions. Réponds avec les informations préparées en Phase 1.

STOP : Si le script s'arrête en erreur, lis le message, note-le exactement, et appelle le Tech Lead.

2.4 — CORRECTION MIGRATIONS (étape manuelle obligatoire)

ATTENTION : Le projet n'a pas encore de fichiers de migration Prisma. Avant de continuer, exécute cette commande manuellement. 
cd /opt/helpdesk/server
npx prisma migrate dev --name init

Cette commande crée les fichiers de migration et applique le schéma à la base. Elle ne sera faite qu'une seule fois.

Ensuite, commite ces fichiers sur main :

cd /opt/helpdesk
git add server/prisma/migrations/
git commit -m "chore: add initial prisma migration"
git push origin main

2.5 — PURGE DE LA BASE DE DONNÉES

DANGER : Cette commande efface TOUTES les données et recrée la base from scratch avec le seed. À n'exécuter QU'UNE SEULE FOIS, avant la mise en service. Jamais en prod avec de vraies données. 
cd /opt/helpdesk/server
npx prisma migrate reset --force

VÉRIFICATIONS POST-INSTALLATION
Lance chaque vérification dans l'ordre. Si l'une échoue, STOP.
# 1. PM2 tourne ?
pm2 status
# → "helpdesk-server" avec statut "online" (vert)

# 2. L'API répond ?
curl http://localhost:3001/api/settings/public
# → Du JSON, pas une erreur

# 3. Nginx tourne ?
systemctl status nginx
# → Active: active (running)

# 4. Le frontend est accessible ?
curl -s -o /dev/null -w "%{http_code}" http://localhost
# → 200

# 5. Depuis un navigateur sur ton poste
# → Ouvre http://<IP_SERVEUR>
# → La page de login doit s'afficher

Vérifier le widget Sports et les pièces jointes

# 6. Les endpoints sports répondent ?
curl http://localhost:3001/api/sports/matches
# → Du JSON avec les matchs de la semaine

# 7. Le dossier des pièces jointes matchs existe ?
ls -la /opt/helpdesk/server/uploads/match-attachments/
# → Le dossier doit exister (créé automatiquement au premier upload)
Widget Sports : Le dashboard affiche un widget "Matchs de la semaine" avec 8 compétitions (Top 14, Pro D2, Champions Cup, Challenge Cup, Starligue, Super League, Ligue 1, ELMS). Les données proviennent de sources publiques, aucune configuration supplémentaire n'est requise.
Starligue : Les paramètres seasons_id et key de l'API LNH doivent être mis à jour dans le code source à chaque nouvelle saison.

MISES À JOUR FUTURES
Déploiement des nouvelles versions
ssh root@<IP_SERVEUR>
cd /opt/helpdesk
bash deploy.sh

Ce script fait automatiquement :

  1. git pull — récupère le nouveau code depuis main
  2. npm ci — met à jour les dépendances
  3. prisma migrate deploy — applique les nouvelles migrations
  4. Build du frontend
  5. Redémarre le serveur Node.js via PM2
  6. Recharge Nginx
Durée estimée : 2-5 minutes. Le service est brièvement interrompu au redémarrage PM2.

Nouveaux endpoints API

Les mises à jour récentes ont ajouté les endpoints suivants :

Dossier uploads : Les pièces jointes des matchs sont stockées dans /opt/helpdesk/server/uploads/match-attachments/. Ce dossier est créé automatiquement. Les fichiers sont purgés 6h après chaque match. Vérifiez que le dossier uploads/ dispose de permissions d'écriture pour l'utilisateur Node.js.

PLAN DE ROLLBACK
Si après un déploiement l'application ne fonctionne plus
cd /opt/helpdesk

# 1. Voir le commit en prod
git log --oneline -5

# 2. Revenir au commit précédent
git checkout <COMMIT_HASH_PRÉCÉDENT>

# 3. Rebuilder
npm ci --prefix server --omit=dev
npm ci --prefix client
npm run build --prefix client

# 4. Redémarrer
pm2 restart helpdesk-server --update-env
Note : Les migrations Prisma ne se rollback pas automatiquement. Si tu as appliqué une migration cassée, appelle le Tech Lead.
